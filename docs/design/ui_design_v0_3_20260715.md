# Nutrition App UI Design v0.3 追補 — 記録タブのショートカット（P2）

v0.1 §8 未決事項「『いつもの食事』の保存単位とデータモデル」を解消し、
「不足を補う候補」の実装仕様を確定する追補。v0.1 / v0.2 の他の節はそのまま有効。

## 1. いつもの食事 = 記録履歴からの自動導出（ユーザー決定 2026-07-15）

新しい永続エンティティは**作らない**。過去の記録（MealRecord）だけから導出する
読み取り専用ビューとする（「記録からの事実」原則と一致）。

仕様:

- 対象: 直近 **14日** の記録のうち、現在選択中の食事区分（breakfast 等）と同じもの
- 集計: food_id ごとの出現回数。**2回以上** 出現した食品を頻出とみなす
- 表示: 出現回数の多い順に **上位4件**。分量は **その食品を最後に記録した時の g**
- 行の構成（6a準拠）: 食品名 + g + 推定kcal + 「＋」（1タップで作成中リストへ追加）
- 履歴が無い/頻出が無い場合はセクションごと非表示（空状態を作らない）
- data model v0.1 への追記は不要（派生読み取りビューのため）。
  手動保存（お気に入り）は将来の拡張とし、必要になった時点で FavoriteItem を追補する

## 2. 不足を補う候補（RecommendationCandidateService）

recommendation boundary（handoff 15）準拠。

- **根拠 context**: 当日の日次分析（GET /api/analysis の同一計算）に紐づける。
  当日の記録が無い日・区分未設定時は候補を出さない
- 対象: 日次サマリーの不足上位 **2栄養素**。各栄養素につき候補 **2食品**
- 候補の選定: FoodMaster 40食品のうち、対象栄養素の含有量（数値のみ）が多い順。
  提示分量は UnitConversion の代表量（数値がある場合）、なければ 100g
- 行の構成（6a準拠）: 食品名 + 分量表示 + 推定kcal +
  「◯◯不足分の約◯%」+ 「＋追加」
- **必須注記**（セクション直下・省略禁止）:
  「出典: 食品成分表(八訂)。含有量に基づく候補で、摂取の推奨ではありません」
- 禁止: 効果保証・治療/予防表現・サプリ提案・疾患別提案（boundary §5）。
  文言は事実（含有量・充足割合）のみで構成する

## 3. 未決事項の更新

v0.1 §8 のうち「『いつもの食事』の保存単位とデータモデル」を本追補で解消
（自動導出・新規エンティティなし）。残りの未決 3 件（折りたたみ展開 UI・
オンボーディング本設計・ユーザー名表示）は該当実装前に追補する。
