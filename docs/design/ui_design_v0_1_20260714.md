# Nutrition App UI Design v0.1

## 1. 目的

デザイン探索（Nutrition UI Explorations）の最終合意案（6a/6b/6c + 5a セマンティクス）を、
本リポジトリの実装単位（`src/features` / `src/components` / `src/app`）へ落とし込む UI 設計の正本。

設計探索の採用結果:

- **IA**: 6a — 3タブ構成（ホーム・記録・分析）。検索はタブから廃止し「記録」内の手段に統合
- **状態表現**: 5a — カラー×状態セマンティクス（全画面共通・1色=1意味・冗長符号化）
- **不足の見せ方**: 5b — 達成先出し、不足は上位2件+折りたたみ、数値は「あと◯◯」に一本化
- **超過の見せ方**: 6b — 「目標量を上回る」事実提示のみ。DG（目標量）と UL（耐容上限量）を区別
- **空状態**: 6c — 初日0件は「最初の1食」に集中。欠食日は分析から除外し理由を明示

## 2. デザイントークン

### 2.1 カラーパレット（07 Recommended UI Palette）

| token | value | 用途 |
|---|---|---|
| `--color-base` | `#FFFDFC` | 画面背景 |
| `--color-surface` | `#DDF7F5` | 淡い面（カード・セクション背景） |
| `--color-primary` | `#2F8C7E` | CTA・選択状態・進捗・摂取済み表示 |
| `--color-primary-deep` | `#206B60` | hover・超過分の濃緑 |
| `--color-accent` | `#F4C23A` | **達成バッジ・チップ限定**（バー/リングの塗りに使用禁止） |
| `--color-text` | `#202A2C` | 本文テキスト |
| `--color-subtext` | `#6A7776` | 補助テキスト |

### 2.2 タイポグラフィ

| 用途 | フォント |
|---|---|
| UI 全般 | Noto Sans JP (400/500/700) |
| 主要数値（あと◯◯・充足率など） | Shippori Mincho B1 (600/700) |

### 2.3 カラー×状態セマンティクス（5a・全画面共通ルール）

1色=1意味。状態は「色+形（✓・斜線・数値）」の冗長符号化で伝え、色覚特性に依存しない。

| 表現 | 意味 | 用途 / 禁止事項 |
|---|---|---|
| 緑（primary）塗り | 進捗・充足 | 摂取済みバー・リング・選択・CTA。充足率にかかわらず「摂った分」は常にこの色 |
| 濃緑（primary-deep）塗り | 100%超過分 | 基準を超えた量の事実表示。良し悪しの判断は付与しない（ゴールドは誤読のため不使用） |
| 斜線パターン（グレー） | 残り必要量 | 基準までの差。色ではなくパターンで表現 |
| 濃色強調（赤は不使用） | 大きな不足 | 充足率70%未満のバー・数値。警告色（赤）は医療的示唆を避けるため使わない |
| 半透明・点線枠 | 追加プレビュー | 保存前の「この食事で増える分」。確定値（緑）と未確定を区別 |
| ゴールド（accent）+ ✓ | 達成 | バッジ・チップ限定。バーやリングの塗りには使わない |
| 淡い面（surface） | 情報・提案 | 文章による事実・提案のコールアウトのみ。データ表現には使わない |

### 2.4 数値・密度・操作の共通ルール

- 数値は **主表現1つ（あと◯◯）+ 補助1つ（摂取/基準）** まで。% とバーの併記は片方に絞る
- 不足の提示は **達成・良好を先に** 見せてから（不安を煽らない）
- 不足リストは **上位2件 + 折りたたみ**。1画面1つの問いに絞る
- タップ対象は **実機44px以上**
- すべての分析表示から **該当日の記録へ1タップ** で遷移できる

## 3. IA（情報アーキテクチャ）

### 3.1 タブ構成（3タブ）

| タブ | 責務 | 主コンテンツ |
|---|---|---|
| ホーム | 今日の事実 | 今日のサマリー（達成先出し→不足上位2件）・不足カード→記録への導線 |
| 記録 | 食事の記録・検索 | 食品検索（タブ内の手段）・いつもの食事ワンタップ・不足を補う候補・追加プレビュー・保存 |
| 分析 | 基準値比較 | 日次の基準値比較・超過表示（DG/UL）・週次レポート・ヒートマップ |

- 食品の栄養値を「調べるだけ」の利用は記録タブの検索結果で完結（追加せず閉じる）
- ホームの不足カード→記録タブ、分析の各行→該当日の記録、へそれぞれ1タップ遷移

### 3.2 ルーティング（`src/app`）

```text
/            → ホーム（今日のサマリー）
/meals       → 記録（検索・いつもの・作成中の食事）
/analysis    → 分析（日次比較・週次レポート）
```

既存の `/foods` ページは独立画面としては作らない（検索は記録タブ内のコンポーネント）。
`/api/foods` は検索データ供給として維持。

## 4. 画面仕様

### 4.1 ホーム（今日のサマリー）

構成（上から順に）:

1. 日付ヘッダ + 「食事摂取基準(2025)との比較」の出典表記
2. **できていること**（達成先出し）: 「9項目中6項目が80%以上」+ 達成バッジ（✓ + accent）上位、「目標圏内 n項目」
3. **足りていないもの**（不足が大きい順・上位2件）:
   - 栄養素名 + 摂取/基準（例 `3.8 / 8.5 µg`）+ 主表現「**あと 4.7 µg**」（Shippori Mincho 大）
   - 食品換算チップ（例「鮭 約20g分 ＋記録」）→ タップで記録タブへ（食材候補は参考情報。推奨表現にしない）
4. 「他の不足 n件 — ◯◯ あと◯◯ ▾」折りたたみ
5. 「今日の記録を確認する →」導線
6. 出典フッタ: 食品成分表(八訂)・食事摂取基準(2025)

空状態（初日0件・6c）:

- 「最初の1食を記録すると基準値との比較が始まります」+ 単一CTA「最初の食事を記録する」
- 「このアプリができること」3点（成分表で自動計算 / 基準との比較 / 不足量の食品換算）
- 免責: 「事実の提示のみを行い、診断・助言はしません」

### 4.2 記録（meal-entry + food-search 統合）

構成:

1. 日付・食事区分ヘッダ（例「夕食を記録」）
2. **検索フィールド**（「食品名で検索 — ここから探して追加」）: FoodMaster 40食品から候補検索
3. **いつもの食事**: ワンタップ追加（品名 + g + kcal + 「+」）
4. **不足を補う候補**: 今日の不足栄養素に基づく FoodMaster 候補
   - 例「鮭の塩焼き 1切れ 146 kcal · ビタミンD不足の90%分 ＋追加」
   - 必須注記: 「出典: 食品成分表(八訂)。含有量に基づく候補で、摂取の推奨ではありません」
5. **この食事に追加済み**（作成中リスト）: 品目・合計kcal、追加プレビュー表現（5aの半透明/点線）
6. 保存CTA: 「夕食を保存(2品 · 306 kcal)」

制約:

- 分量入力は **explicit grams first**（自動 piece/cup 換算はしない）。「1切れ」等の表示は UnitConversion seed の代表量に基づく表示のみ
- 候補提示は AnalysisRun 等の根拠 context に紐づける（recommendation boundary 準拠）

### 4.3 分析（reference-comparison + daily-summary 週次）

**日次: 基準値比較**

- reference_type 別に表示を分ける:
  - RDA/AI 系: バレットバー（摂取済み=緑、残り=斜線）+ 「摂取/基準/あと◯◯」
  - **DG（目標量）超過（6b）**: 「目標量を上回っているもの」セクション。事実のみ提示
    - 食塩相当量: 「目標 6.5g未満/日 → 8.2g（目標より +1.7g）」+ 内訳上位の食事
    - 脂質（%E）: 範囲バー（網掛け=目標範囲 20〜30%E、濃緑=超えた分）
    - 必須注記: 「目標量(DG)は生活習慣病予防のための基準で、超過=直ちに問題を意味しません。耐容上限量(UL)が設定された栄養素が上限に達した場合のみ、別途明示します。判断・助言は行いません」
  - UL 到達時の表示は未設計（→ §8 未決事項）
- 「内訳になった食事を確認 →」で該当日の記録へ

**週次: 充足率レポート / バランスマップ**

- 曜日×栄養素ヒートマップ（5c）: セル内は数値(充足率%) + ✓(達成) + 斜線(50%未満) の冗長符号化
- 「今週のパターン（記録からの事実）」: 例「ビタミンD: 7日中5日で50%未満 → 記録を見る」
- 欠食日の扱い（6c）: 記録なし日は平均から除外し、理由を明示
  （「0kcalとして扱うと平均が実際より低く出るため」）+ 「あとから追加 →」導線

## 5. Safe Wording / 免責の画面配置

health-safety-wording-policy v0.1 準拠。UI 文言はすべて SafeWordingService（`src/domain`）を通す。

| 画面 | 必須表示 |
|---|---|
| オンボーディング（初回） | 注意事項と同意（推定値・医療の代替でない旨） |
| ホーム / 分析 | 「出典: 食品成分表(八訂)・食事摂取基準(2025)」+ 推定値である旨 |
| 記録の候補提示 | 「含有量に基づく候補で、摂取の推奨ではありません」 |
| 超過表示 | DG/UL の説明と「判断・助言は行いません」 |

- 禁止: 断定（「不足です」）・警告色（赤）・効果保証・医療的指示
- 使用: 「不足傾向」「目標量を上回る」「候補」「参考情報」等の事実・傾向表現

## 6. 実装マッピング

### 6.1 features / components

```text
src/features/daily-summary/          ホーム画面（達成先出し・不足カード・空状態）
src/features/meal-entry/             記録画面（いつもの・追加済みリスト・保存）
src/features/food-search/            記録タブ内の検索（画面ではなくコンポーネント群）
src/features/reference-comparison/   分析画面（日次比較・超過表示・週次レポート）

src/components/                      再利用UI
  BulletBar          摂取済み(緑)+残り(斜線)+超過(濃緑) のバー
  RangeBar           DG範囲バー（網掛け範囲+超過分）
  AchievementBadge   ✓+accent の達成バッジ（accent使用はここに限定）
  RemainingCard      「あと◯◯」主表現カード + 食品換算チップ
  HeatmapCell        数値+✓+斜線 の冗長符号化セル
  TabBar             3タブナビゲーション
  SourceFooter       出典・免責フッタ
  EmptyState         空状態（初日・欠食日）
```

- 栄養計算・判定・文言生成は `src/domain`（NutrientCalculation / ReferenceJudgment / SafeWording / RecommendationCandidate）。features には置かない
- デザイントークンは `src/styles/tokens.css`（CSS custom properties）に一元定義

### 6.2 画面 ↔ API

| 画面 | API |
|---|---|
| ホーム | `GET /api/analysis`（当日サマリー: 充足率・不足量・食品換算） |
| 記録・検索 | `GET /api/foods`（FoodMaster検索）/ `POST /api/meals`（保存） |
| 分析 | `GET /api/analysis`（日次・週次。欠食日除外は domain 側で処理） |

API 契約の詳細は `05_api_design_v0_2` と OpenAPI YAML（作成待ち）に従う。route.ts は薄く保ち、実処理は `src/server/api/handlers`。

## 7. 実装順序（MVP 内の優先度）

handoff の First task（seed read model → validation tests）完了後:

1. **P0**: デザイントークン + TabBar + 記録画面（検索・g入力・保存）— 記録がないと他画面が成立しない
2. **P0**: ホーム（日次サマリー・空状態）— BulletBar / RemainingCard / AchievementBadge
3. **P1**: 分析・日次（基準値比較 + 6b 超過表示）
4. **P1**: 分析・週次（充足率レポート → ヒートマップ）
5. **P2**: いつもの食事ワンタップ・不足を補う候補（RecommendationCandidateService 依存）

デスクトップ（1c/2a/4c 案）は MVP 対象外。モバイル優先・レスポンシブで崩れない範囲に留める。
