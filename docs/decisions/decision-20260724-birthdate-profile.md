# decision-20260724: 基準の区分を生年月日から自動判定する

- Status: accepted
- Date: 2026-07-24
- Context: MVP では年齢帯（adult_18_29 等）をユーザーが手で選ぶ方式だった。
  誕生日を過ぎても比較先が古い区分のまま固定され、ユーザーが気づいて設定を
  変えない限り誤った基準と比較し続ける。

## 決定

1. 保存する正本は **生年月日（YYYY-MM-DD）× 性別**。年齢帯は保存しない。
2. 年齢帯は **評価対象日ごとに導出**する（`resolveProfileForDate`）。
   「今日」ではなく分析している日を基準にするため、過去日はその日に
   適用されていた区分のまま評価される。週次・月間の庭も日ごとに解決する。
3. **18歳未満・120歳超は 422 で拒否**する。凍結seedに収録している
   食事摂取基準(2025)は成人（18歳以上）の行のみで、近い区分での代替比較は
   「その人の基準ではない値」との比較になるため行わない。
4. 旧プロフィール（`ageBand` 固定値）は**そのまま読めて動く**。生年月日が
   入力された時点で自動判定へ切り替わる。バックアップの復元も両形式を受理する。

## 影響

| 層 | 変更 |
| --- | --- |
| domain | `reference/ageBand.ts` を新設（`ageOn` / `resolveAgeBand` / `isValidBirthDate`）。既存の判定ロジックは不変 |
| server | `profileResolution.ts` を新設。日次・週次・庭・候補の4ハンドラが日付ごとに解決 |
| store | `StoredProfile` = `{ sex, birthDate? , ageBand? }`。書き込みは birthDate を優先 |
| API | `PUT /api/profile` の必須項目が `ageBand` → `birthDate`。エラーコードに `invalid_birth_date` / `age_below_supported_range` / `age_above_supported_range` を追加。`GET` は表示用に導出 `ageBand` を同梱 |
| UI | オンボーディングの年齢帯ボタン → 日付入力（判定結果をその場に表示）。分析タブに設定セクション（`ProfilePanel`）を追加し、後から変更可能に |
| seed | **変更なし**（公式値には一切触れない） |

## 意図的に採らなかった選択肢

- **年齢を数値で入力**: 1年ごとに手直しが必要で、元の問題が残る
- **保存時に年齢帯を計算して保存**: 保存した瞬間から古くなる。導出は読み取り時にのみ行う
- **18歳未満を最も近い区分で代替**: 事実と異なる比較になる。safe wording policy に反する

## 未決

- 女性の鉄は月経の有無で公式値が分かれる（seedは条件付きのまま保持）。
  現状は `unknown (conditional_value)` として判定しない。設定項目として
  持つかは、機微情報であるため別途ユーザー判断を要する。
